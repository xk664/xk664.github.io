# 缓存

**==各种缓存问题的核心：大量请求到达数据库，造成数据库崩溃==**
**好处:通过将数据缓存再Redis(内存)，降低了对数据库的访问（磁盘）**

## 缓存更新策略

- 先修改数据库再删除缓存为什么优于先删除缓存，再修改数据库？

如果线程1此时先删除缓存，再修改数据库的时候，由于线程1删除了缓存所以线程2未命中，就会查询数据库，此时由于线程1还未修改完成，线程2就会查询到旧的数据，然后将旧的数据写入缓存，这样后面的每次查询都命中的是旧数据，如果是先修改数据库再删缓存，即使线程2先获取到旧的数据到缓存，后面线程1也会将其删除

## 缓存穿透

- 概念
当很多类似的空请求到达数据库时，会造成数据库崩溃

- 解决方案
方案一：当数据库中也没有数据时，就再redis中缓存一个null值（设置TTL,防止缓存中null太多），当后面又出现这种空请求的时候，就不会到达数据库，防止数据库崩溃
方案二：布隆过滤器

	再查询进入redis以前，先通过布隆过滤器判断，数据是否存在，存在放行到redis,不存在直接返回空数据

	实现：bitmap,每一个id通过计算，算出多个hash至，将数据存在的id的hash对应的bit，设置为1，如果查询时，id对应的hash的值都为一，则说明数据存在

	缺点：存在误判（id2可能计算出来的hash一个是id1的hash,一个是id3的hash,然而这两个id的数据肯定存在，所以会判成数据存在）

	解决：增大数组大小，减少hash冲突

## 缓存雪崩

- 概念
缓存在redis的key再统一时间大量失效，导致大量请求同时访问数据库

- 解决方案
1.再缓存空key的时候，再TTL后面加随机值，使他的失效时间，不是聚集再同一时刻，而是分散再一个时间段
2.Redis集群
3.给缓存业务添加降级限流策略
4.添加多级缓存

## 缓存击穿

- 概念
缓存击穿也叫做热点key问题，指的是一个被高并发访问的key突然失效，一瞬间大量请求涌入数据库
- 解决方案

![](/images/notes/Screenshot 2025-11-10 114633.png)

**互斥锁**

当热点key过期后，会高并发访问数据库，此时我就只让一个线程去访问数据库（互斥锁），并更新缓存，然后其他线程则因为无法获取锁，则等待，等到缓存更新完，释放锁后，其他线程则会读取到缓存，就不会继续访问数据库了

缺点：线程需要等待，性能较低；可能有思索风险
有点：保证了一致性；实现简单；没有额外内存消耗

**逻辑过期**

多增加一个字段expire表示过期时间，当线程进来发现Redis里面的这个key过期了，就会试图获取互斥锁，获取成功的线程，会开辟一个新的线程，去访问数据库并更新缓存，然而此时获取到锁的这个线程会将锁交给新开辟的线程，原来的线程则会直接返回旧的数据，同理，其他线程发现过期了，也会试图获取互斥锁，然而由于互斥锁已经被另一个线程获取，所以其他锁获取失败，直接返回旧数据

缺点：不保证一致性；有额外内存消耗（expire字段）
优点：线程不用等待，性能好

# 锁

## 悲观锁

- 概念
认为一定会发生线程安全问题，所以再操作数据之前，加锁，保证线程串行执行

缺点：性能低
优点：安全
## 乐观锁

- 再操作数据之前，检查先检查数据是否有变化

- 解决方法

1.版本号法（CAS）
给要操作的数据加一个版本号，每进行一次操作就修改版本号，线程1再操作前先获取版本号，再即将要操作数据的时候再次获取一次版本号，将如果两次版本号一直，则说明，再之间，没有其他线程进行数据操作，所以此时我就可以放心操作数据，如果版本号不一样，就得另外再次查询数据是否符合条件，再来判断是否操作

## 分布式锁

不能用JVM内部的锁，只能用JVM外面的锁，这样多个JVM可以用同一个锁

- 概念

满足在分布式系统或集群模式下多进程可见并且互斥的锁

- 特点
1.高性能
2.多线程可见
3.互斥
4.安全性
5.高可用

### Redisson
封装了分布式锁等一系列功能
- 看门狗

每设置一把锁都会设置过期时间，为了防止因为线程崩溃导致锁一直无法释放，但是如果本身业务执行时间就超过了上设置的过期时间，也会提前释放锁，出现并发问题，所以看门狗就会自动续期（刷新有效期），防止这种情况的发生

- 不可重入问题解决

概念：同一个线程不能多次获取同一把锁

解决方法(可重入锁)：通过多加一个字段，标识获取锁的次数，当一个线程试图获取锁的时候，先判断是否时自己的锁，如果是，则字段加一，不是则启动尝试并等待，释放锁时就先判断标识是否小于等于0，使得话就释放锁，不是就标识减一

- 主从一致性
概念：redis集群中，主节点崩溃导致数据（锁）没有及时同步到从节点，从而数据（锁）丢失

解决方法(联锁):多个独立的redis节点，没有主从关系，获取锁时，只有每个节点都获取到才算获取成功,加锁时，也是只有都成功才成功，释放锁时就是释放每个人节点的相同的锁

# 分布式缓存

## 持久化

### RDB

主动停止运行时，会自动执行save

- 概念
将redis内存里面的内容写入磁盘，实现持久化

- 命令
save--->redis的主线程会将内存的内容加载到磁盘，由于Redis是单线程，主线程已经被占用，那后面对redis操作的线程都将会阻塞
bgsave--->会单独开启一个子进程去加载数据到磁盘，不会阻塞

![](/images/notes/Screenshot 2025-11-19 205827.png)

当在save时，如果主进程此时有写操作，会先拷贝一份原来的数据，在拷贝的数据上做写操作，然后再合并

### AOF

- 记录每一次命令的文件

- 每次启动时会执行AOF的命令，实现数据恢复

## Redis主从

**优点：可解决高并发的读操作，以及数据同步**
**缺点：无法解决高并发的写操作，因为只有一个master，然而只能再master执行写操作；海量数据无法保证，因为每一个节点都会储存海量数据**
### 搭建主从结构

```shell

slave 主节点IP 主节点端口

config set masterauth 主节点密码

# 查看节点的从节点情况
info replication
```

搭建完成后，主节点的数据会自动同步到从节点

==如果再这之前从节点有数据，则会先删除自己的数据，再加载RDB文件==
### 全量同步原理（第一次同步RDB）

![](/images/notes/Screenshot 2025-11-20 112622.png)

**replication id: 每个节点都有自己的id**
**offset:偏移量，用于判断从节点和主节点的同步进度（相当于数组的下标）标记repl_baklog**

**repl_baklog:再同步期间执行的写命令**

- 如何判断时第一次同步？
因为只要同步后，主节点会将自己的replication id发送给从节点，如果他们的id不相同，说明还没有同步，是第一次同步


### 增量同步原理（用于slave重启后）(同步储存再缓冲区的命令非AOF)

![](/images/notes/Screenshot 2025-11-20 113802.png)

如果从节点重启，再这期间主节点还在进行操作，那么主节点的offset就会增大，而从节点不变，此时重启后，从节点只需要从他的offset开始，同步到主节点的offset(repl_baklog)

- 如果主节点在期间进行了太多操作，导致覆盖了之前从节点还没有备份的数据怎么办？

导致数据丢失，恢复不了，所以应该尽量去避免，并且会触发全量同步

### 优化

- 再master的配置文件配置repl-diskless-sysnc yes  这样同步数据时，直接通过网络发送，而不是先写入磁盘 ，再发送（减少IO）

- 减少Redis单节点的内存，减少RDB导致的过多磁盘IO

- 提高repl_baklog的大小，尽可能避免全量同步

- 限制master的slave数量，多的slave加到slave的slave形成主--从--从链式结构，从而减轻matser压力

### Redis哨兵（sentinel）

- redis客户端，会先连接你指定的节点，然后每隔一段时间通过发送**CLUSTER NODES` 或 `CLUSTER SLOTS命令来获取整个集群的结构，获取最新的集群结构和master

- 作用
1.监控redis集群，如果master宕机，就立即启动slave代替master,并将集群结构返回给客户端
2.如果slave宕机，就立即重启slave

### 搭建哨兵集群



#### 监控机制

哨兵每个每隔一秒，向redis集群，发送ping命令，如果某一个哨兵，没有收到响应，则认为主观下线（实际可能没有）
当超过指定数量（quorum）的哨兵都没有收到响应，就认为这个节点客观下线了

#### 选举机制

 - 先判断slave节点和master的断开时间，如果超过指定值（down_after_milliseconds*10)就排除这个节点

- 判断slave的slave_priority值，越小优先级越高，为0则派出

- 判断offset的大小，越大说明数据越新，优先级越高

- 最后如果上述都相同，就选择runId小的

#### 故障转移

- 给选定的salve发送slaveof no one(告诉他，他现在是master)

- 再给其他slave发送slaveof 新的masterIP  port

- 再修改刚刚宕机的主节点配置文件为slaveof ........,这样他重启后就会加入集群成为slave

## 分片集群结构

**特征：集群有多个master,每个master保存不同数据，每个master又可以形成主从结构**

**监控：master之间通过ping相互监控**

#### 搭建集群

启动每个redis后
```shell
redis-cli --cluster create --cluster-replicas 1 192.168.232.128:7001 192.168.232.128:7002 192.168.232.128:7003 192.168.232.128:8001 192.168.232.128:8002 192.168.232.128:8003
```

--cluster-replicas 1 代表每个master有一个slave,排在前面的就是master
### 散列插槽

**理解:类似于hash，每个key都有对应的hash值**

**大小：16384（0~16383）**

**作用：分片集群是每个master存的数据都不相同，那我怎么知道每个数据该存在那个节点，创建集群后，每个master都会平均分配这16384个插槽，这样key计算出来的hash值(CRC16算法再%16384)，处于那个节点的范围，就存在那个节点，取数据也是这样**

**计算规则：如果指定key的时候，key有{}，则用{}的数据计算，如果花括号里面没有数据或者没有花括号，那么以整个的字符串（包括花括号）计算**

### 集群伸缩（添加节点/删除节点）

#### 添加节点

- 如果不指定是slave,默认为master

- 添加后不会分配插槽的

- 手动分配插槽
```shell
redis-cli --cluster reshard 被分配节点的ip:port
```

#### 删除节点

- 从节点
直接删除

- 主节点
先迁移插槽（数据）到其他master，然后先删除从节点，最后删除主节点
