#  内存结构

## 程序计数器(寄存器)

- 作用：
记录下一个jvm指令的执行地址，当多线程并发时，比如线程1正在执行，然后后面又到了线程2执行，此时程序计数器可以帮助记住线程1运行的下一条指令的地址

- 特点
1.线程私有的，每个线程有自己的程序计数器，随线程的创建而创建，随线程的销毁而销毁
2.不存在内存溢出

## 虚拟机栈

- 概念
1.每个线程运行时需要的内存（可通过 -xss 修改）
2.由多个栈帧组成，一个栈帧就是一次方法调用所需要的内存（局部变量，返回值....）,调用方法时入栈，调用完出栈（释放内存）
3.每个线程只能有一个活动的栈帧，代表此时正在执行的方法（栈顶栈帧）

- 栈内存溢出（栈的大小固定）
1.由于栈帧过多（递归太深），导致溢出
2.栈帧过大，导致溢出

## 本地方法栈

- 用于存放本地方法的调用（不是由java实现的方法，而是用其他语言实现的方法，用native修饰的）

## 堆
- 通过new关键字，创建对象都会使用堆内存（可通过-xmx 修改）
- 特点
1.有垃圾回收机制
2.线程共享，需要考虑线程安全问题

- 查看方法
1.命令行输入jps查看进程号
2.在输入jhsdb jmap --heap --pid 进程号（JDK9+）
JDK9一下用jmap -heap 进程号

或者直接用jconsole图形化

## 方法区

JDK8以后，储存在元空间（metaspace）,所用的时本机内存
JDK8以前，在JVM内存


### StringTable(哈希表)

- 常量池
他是字节码文件的静态池，每个字节码文件都有自己的常量池（储存字符串常量，final修饰的常量等）
- 运行时常量池
有常量池的常量加载而来，此时常量池中的常量就会指向运行常量池的常量，当运行常量池中已经存在了常量池要加载的常量，此时常量池的就会复用常量
- 字符串常量池（哈希表）
JDK6及以前位于方法区，以后位于堆区
- ==流程==
1.先编译要运行的类
2.在加载到内存
3.在运行代码

==注：==

 1.类加载阶段：字符串常量仅进入 “运行时常量池”，未进入 “字符串常量池”

当类被加载时：

- JVM 会将 Class 文件常量池中的字符串字面量（如`"abc"`）加载到**运行时常量池**，此时存储的是字符串的 “符号信息”（如 UTF-8 编码的字节序列），属于 “符号引用”，还未关联到具体的字符串对象。
- **此时字符串常量池（String Pool）中还没有这个字符串对象**，只有当代码首次使用该字符串时，才会触发后续的入池逻辑。

2. 代码首次使用时：字符串常量才进入 “字符串常量池”

当代码执行到引用该字符串的语句（如`String s = "abc"`）时：

- JVM 会解析运行时常量池中的符号引用，检查**字符串常量池**中是否已存在内容为`"abc"`的对象。
    - 若存在：直接复用该对象，运行时常量池将符号引用替换为指向该对象的 “直接引用”（地址），变量`s`指向这个地址。
    - 若不存在：在字符串常量池中创建`"abc"`对象，运行时常量池记录其地址，变量`s`指向新对象。

```java
	String s1="a";
	String s2="b";
	String S3=s1+s2;
	String s4="a"+"b";
	
	
	s4.intern()
	//jDK1.8以后会将这个字符串放入串池，有就不放，没有则放入，返回的时串池中的对象，堆区的就不存在了，JDK1.6会将s4拷贝一份在放入串池，s4依旧指向的堆区的
```
s1和s2都在串池中
s3:new StringBuilder().append("a").append("b").toString();-->最后在堆区创建了一个String
s4:编译器会优化为"ab",如果串池中有，则复用，反之则加入串池

## 直接内存

- 特点
1.相对于io读取从磁盘中读取数据更快
2.不受java的垃圾回收管理
3.不是由java的垃圾回收释放内存，而是ByteBuffer底层的unsafe的freeMemory()，当ByteBuffer(内部使用了Cleaner‘虚引用'来检测ByteBuffer)被java的垃圾回收回收时，会由一个ReferenceHandler线程通过cleaner的clean方法调用freeMemory,防止内存没有释放
