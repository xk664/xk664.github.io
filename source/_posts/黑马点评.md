# 疑问？

- ==为什么ThreadLocal在一个请求后还要清空数据，线程销毁后数据不应该自动销毁了吗？==

web应用中使用的连接池,一个请求结束后线程不会立即销毁，而是回到连接池，此时如果另一个用户登录，而也使用了这个线程，可能会读取到前一个用户的数据，所以要销毁
```java
// ThreadLocal 风险场景
// 请求1：用户A登录
UserHolder.saveUser(userA);  // ThreadLocal存储用户A

// 线程复用处理请求2：用户B的请求
// 如果不清理，UserHolder中仍然是用户A的数据
// 可能导致用户B看到用户A的信息

```

- ==选择session的好处？==

用户隔离：每个用户的 HttpSession 是独立的
天然隔离：不同用户的 session 数据不会相互访问
生命周期明确：session 数据绑定到特定用户会话

- ==为什么后面用redis代替session?==

当tomcat服务器有多个的时候，服务器键间不能共享session里面的数据，导致本来登录了，却显示未登录等问题，所以采用redis代替session（多台服务器间可以共享数据）

- ==为什么LoginIntercepter中的StringRedisTemplate只能通过构造器注入？==

构造器注入，因为在config里面是直接new的这个拦截器，不是有spring创建（有@component之），所以不能用@Autowired
```java
//构造器注入，因为在config里面是直接new的这个拦截器，不是有spring创建，所以不能用@Autowired  
private StringRedisTemplate stringRedisTemplate;  
public LoginInterceptor(StringRedisTemplate stringRedisTemplate) {  
    this.stringRedisTemplate = stringRedisTemplate;  
}

@Configuration  
public class MvcConfig implements WebMvcConfigurer {  
    @Autowired  
    private StringRedisTemplate stringRedisTemplate;  
    @Override  
    public void addInterceptors(InterceptorRegistry registry) {  
        registry.addInterceptor(new LoginInterceptor(stringRedisTemplate))  
                .addPathPatterns("/**")// 默认  
                .excludePathPatterns(  
                        "/user/code",  
                        "/user/login",  
                        "/blog/hot",  
                        "/shop/**",  
                        "/shop-type/**",  
                        "/upload/**",  
                        "/voucher/**"  
                );  
    }  
}
```

- ==先修改数据库再删除缓存为什么优于先删除缓存，再修改数据库==？

如果线程1此时先删除缓存，再修改数据库的时候，由于线程1删除了缓存所以线程2未命中，就会查询数据库，此时由于线程1还未修改完成，线程2就会查询到旧的数据，然后将旧的数据写入缓存，这样后面的每次查询都命中的是旧数据，如果是先修改数据库再删缓存，即使线程2先获取到旧的数据到缓存，后面线程1也会将其删除

- ==如何解决缓存穿透问题?(当请求的数据既不在缓存中也不再数据库)==

当很多类似的空请求到达数据库时，会造成数据库崩溃
方案一：当数据库中也没有数据时，就再redis中缓存一个null值（设置TTL,防止缓存中null太多），当后面又出现这种空请求的时候，就不会到达数据库，防止数据库崩溃
方案二：布隆过滤器

- ==已经加了synchronized对userid上锁，保证了一户一单，但锁里面的乐观锁（防止多个用户同时下单，超库存）是否可以删除了？==

有用，因为synchronized只是锁住了同一个用户的多个线程，然而当多个不同用户同时抢券是，synchronized不起作用了，所以乐观锁依旧起作用

- ==为什么再解决一户一单的问题时，是锁住方法，而不是方法里面的购买部分？(单体项目)==

@Transactional是基于Aop实现，执行流程：先开启事务-->执行方法逻辑-->提交事务（不是方法执行完就提交了事务),不是return就代表方法执行完了，还要执行Aop(提交事务),方法执行完成，代表事务也提交了，不然会存在事务还未提交（订单还没有创建）就释放锁了

- ==一户一单问题（集群）?==

Redis实现的分布式锁，判断是否购买是MySQL实现，所以Redis的锁用完后应该释放

- ==为什么使用Redisson?==

他是一个组件，已经封装了分布式锁，并且解决了误释放锁，锁超时等等操作，以及其他功能，便于开发，提高开发效率

- ==异步秒杀优化？==

开始有一个线程执行（库存是否足够->之前是否购买过->操作数据库），非常耗时，一次业务就非常耗时，聪明的你想到，将这个流程交给多个线程去执行，一个线程负责检查是否符合购买条件，另一个负责数据库操作，如果符合条件，则直接将订单id等先返回给用户，并且将这些信息进入阻塞队列，另一个线程在从队列里面获取信息，执行相关数据库操作，这样第一个线程在判断完之后就可以立马执行下一个业务，将待操作数据库的任务push到队列即可

- ==lua脚本的好处?==

将多个操作合成一个，实现原子性，避免了一户多单问题
# 理解

1.LoginIntercepter只是定义了拦截器拦截到==一个请求==后如何操作，pre是处理请求前，after是后
```java
public class LoginInterceptor implements HandlerInterceptor {  
    @Override  
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {  
        HttpSession session = request.getSession();  
        User user =  (User)session.getAttribute("user");  
        if(user == null){  
            response.setStatus(401);  
            return false;  
        }  
        UserDTO userDTO = new UserDTO();  
        BeanUtils.copyProperties(user, userDTO);  
        UserHolder.saveUser(userDTO);  
        return true;  
    }  
  
    @Override  
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {  
        UserHolder.removeUser();  
    }  
}
```
2.拦截器具体的作用范围（用于拦截哪些请求）需要到配置类编写
```java  
@Configuration  
public class MvcConfig implements WebMvcConfigurer {  
    @Override  
    public void addInterceptors(InterceptorRegistry registry) {  
        registry.addInterceptor(new LoginInterceptor())  
                .addPathPatterns("/**")// 默认  
                .excludePathPatterns(  
                        "/user/code",  
                        "/user/login",  
                        "/blog/hot",  
                        "/shop/**",  
                        "/shop-type/**",  
                        "/upload/**",  
                        "/voucher/**"  
                );  
    }  
}
```

3.
通过order（），指定拦截顺序，或者先添加的后拦截，后添加的先拦截

4.@Transactional是基于Aop实现，执行流程：先开启事务-->执行方法逻辑-->提交事务（不是方法执行完就提交了事务),不是return就代表方法执行完了，还要执行Aop(提交事务),方法执行完成，代表事务也提交了

5.
synchronized只能锁住对一个JVM(一个进程)上的线程，如果后面服务器以集群方式分布，则会失效
