## 为什么redis这么块？

1.基于内存
2.单线程，避免了频繁的上下文切换和并发问题
3.I/O多路复用

内存里面的数据需要先到内核缓冲区，再从内核缓冲区拷贝到用户缓冲区
**阻塞I/O:一个socket请求来了，如果内核缓冲区没有数据就一直阻塞
非阻塞I/O:一个socket请求来了，如果内核缓冲区没有数据就直接返回异常并一直尝试（提升不大）
、
上面两个都是只有等到这个socket的数据拷贝成功了再处理下一个socket

**I/O多路复用：同时监听多个socket,那个先由数据，就先处理那个**

- I/O多路复用分类
poll,select,epoll

## 缓存穿透怎么解决？


1.设置空数据再redis

		先将数据预热到redis,等下一次查询如果redis命中，就直接再redis返回，未命中就会查询数据库，如果数据库有，就返回数据，并填写至缓存，没有就添加一个空数据到redis,下次来查询时，检测到id对应的value为空，就直接返回无数据。

		缺点：没有数据的id非常多，意味着我们要缓存很多空数据，再redis,内存浪费

	为了解决这个问题你又想到，设置TTL过期时间（所有key）-->保证更新数据

	

	会出现的问题：
				1.当再TTL未过期，但是数据库数据已经发生更改，那么就会返回空数据或者旧数据
				2.大量key同时失效，造成缓存雪崩
				3.热点key失效造成缓存击穿

2.布隆过滤器

再进入redis之前，先通过布隆过滤器查看数据是否存在，存在就差redis,redis不存在就插数据库


- 两种方法数据一致性问题：
		通过设置TTL来定时更新数据
	
- 设置TTL好处
	方案1：减少内存消耗，可以定期更新数据
	方案2：定时更新数据
- 坏处：
		1.当再TTL未过期，但是数据库数据已经发生更改，那么就会返回空数据或者旧数据
		2.大量key同时失效，造成缓存雪崩
		3.热点key失效造成缓存击穿

## 缓存击穿？

### 1.互斥锁（setnx）

线程1先未命中redis,然后获取互斥锁，获取成功就去数据库查询数据并更新缓存，而此时其他线程由于线程1获取了互斥锁，且线程1还未更新缓存（释放锁），则其他线程就一直等待，知道缓存更新

缺点：性能低
优点：数据强一致性

### 2.逻辑过期（要求再解决缓存穿透的时候设置的逻辑过期）

线程1先查看数据是否过期（当前时间大于会的逻辑过期时间）,然后获取互斥锁，获取成功就开启一个新的线程去更新缓存，其他线程在后面获取锁失败后，就直接返回旧的数据，避免等待

优点：高性能
缺点：数据一致性无法保障

## 数据一致性的保障

**==统一做法：只有redis未命中，再查询数据库时，才填入缓存，修改数据库不填写缓存==**

错误做法：

- 先删缓存，再更新数据库

线程A删除缓存后，还没来得及跟新数据库，此时线程B再redis未命中，就会读取数据库，但是数据还没有来得及更新，还是旧数据，然后他由于第一次未命中redis,所以会将这个旧数据填写到缓存，而此时数据库和redis的缓存就不一致了

- 先更新数据库，再删缓存

如果再删除缓存的过程中，出现某种问题（网络波动），导致删除不成功，那么缓存就一直是旧数据

- 延迟双删（还是存在数据不一致的情况）

先删除缓存-->更新数据库--->(延迟一段时间)删除缓存）

先看先删除缓存再更新数据库，会导致redis是旧数据，DB是新数据，所以延迟一段时间，等最后大部分更新操作，完成，就删除再次删除缓存（旧数据），这样下次来未命中redis，就会访问数据库，并将数据填写到redis

- 分布式锁（强一致性）

再你要更新数据时，先获取锁，然后删除缓存，更新数据库，==可以采取异步==，后面跟新缓存，最后解锁 ，其他线程则一直等待，知道缓存更新完成

## 分布式锁
#### 单redis

一个线程尝试获取锁，获取成功就执行扣减库存等业务，最后释放锁。
获取锁失败，就阻塞或尝试，直到锁被释放。

**可能出现的问题？**

- 如果获取到锁的线程再执行业务时，服务器宕机，造成无法释放锁，怎么解决？

再获取锁时设置一个过期时间

-  如果业务时间过长，导致锁的过期时间已经到了 ，也业务还没有执行完，怎么办？

redisson的看门狗机制，每隔（releasetime/3）时间，自动续期

- 如果再执行业务时，需要线程再次获取他刚刚已经获取的锁（未释放）怎么办(可重入问题）

redisson的可重入锁（再获取锁时先判断这个锁是不是该线程的，是的话，就+1，当释放锁时，就-1，当这个值等于0时就释放锁）

	HaSh储存，锁名字：{UUID}+{线程ID}:重入次数

- 如何解决误删问题？（没有看门狗机制才会出现）

**线程ID和UUID都不同，为什么线程B会误删线程A的锁?**
删除锁，是删除key,而不是field

redisson底层的释放锁和业务执行和获取锁，都是再一个lua脚本执行，保证了原子性

#### redis集群

redisson锁不能解决，只能用他提供的红锁（或者zookeeper）
红锁：
获取锁：再大多数redis获取成功就成功，且当前时间减去获取锁的开始时间小于锁过期时间

释放锁：再多数redis释放成功就算成功

 ## 为什么innodb选择B+Tree作为索引结构?
1.当用二叉排序树插入数据时，如果数据本身有序，插入和查询的复杂度高
2.红黑树度数最大为2,当数据量大时，树的深度很深，查找效率低
3.hash只支持等值查找，不支持范围查找
4.对于B-Tree 
	 由于非叶子节点既存索引键又存实际数据，单个节点的体积会更大
	 这意味着一次磁盘 IO 能加载的 B 树节点数量更少
	 要定位到目标数据，可能需要访问更多节点，从而触发更多次磁盘 IO
	 B+Tree说有数据都在叶子节点，更适合范围查询

## 页分裂和页合并

- 页分裂
一个页就是一个叶子节点，可以存储多条数据，当这个页满了之后，再插入数据的时候，就会将原来页中的数据排序后，分一半到新开辟的页里面，然后就多多了一个叶子节点，此时也会多一个非叶子节点（即新开辟的页里面编号最小的），如果非叶子节点开辟后，也发生了也分裂，则递归到根节点

- 页合并（降低查询次数）

当一个叶子的数据量远小于页的空间大小，就会看他相邻的页是否由空间（这个页的数据+相邻页的数据<=页大小）就会合并，同时父节点也会对应调整

## 什么是undo log和redo log,有什么区别?

MySQL内存架构
- 缓冲池
缓存着磁盘上经常操作的真实数据，再执行增删改查的操作时，先操作缓冲区的数据，再以一定频率刷新到磁盘，减少了磁盘I/O，加快处理速度


redo log就是记录的每一次写操作，当将缓冲区的数据同步到磁盘的时候，出现错误，此时就可以用redo log恢复数据

- 回滚日志（undo log）
当事务回滚时，用于恢复数据
用于记录修改前的数据，比如我执行了insert语句，他保存的就是delete........,
当事务提交后，不会立刻删除，这些日志还会可能应用于MVCC

redo log保证了事务的持久性，undo log保证了事务的原子性和一致性

# 框架篇
## spring的单例bean是线程安全的吗？

 单例 Bean：容器内唯一实例，所有线程 / 使用者共享同一个；（IOC容器里面只有这一个）
 多实例 Bean：每次获取都 new 新对象，线程/使用者各自持有独立实例。

当这个单例bean的成员都是无状态（只读）就没有线程安全；
**例如：如果此时一个单例bean里面，有一个cnt属性，并且对其++，由于这个是单例bean,每个线程共享这一个bean,就会出现线程安全问题**

## AOP原理？

Spring会先创建一个原始对象，然后根据这个生成一个代理（包含原始对象所有功能，并且还有由AOP的增强逻辑），然后将这个代理对象加入IOC容器并替换到原来的原始对象，后面的用的就是IOC容器里面的代理对象（即使这个对象只有一个方法被增强，也会创建整个对象的代理）

## Transitional实现原理？

利用AOP,在执行方法之前，先开启事务(MySQL)，,try方法，如果出现异常就回滚(默认只回滚Runtime异常)，可通过Transitional(RollBackFor=Exception.class),来表明回滚所有异常
```java
try{
	//开启事务
	//执行方法
	proceed();
}catch（运行时异常）{
	//回滚
}
```

## 事务失效情况？

- 异常捕获失效
transional本身会捕获方法运行时异常，然后回滚事务，如果此时方法内部捕获了这个异常，此时AOP就不会在捕获到了，就不会回滚事务
**解决：在方法的catch里面再抛出一个RuntimeEXception**

- 检查异常失效(比如FileNotFoundException)

Transitional默认只回滚Runtime异常，可通过Transitional(RollBackFor=Exception.class),来表明回滚所有异常

- 非public方法失效

因为事务时用AOP实现，所以涉及到生成代理对象
	JDK动态代理：代理对象实现原始对象实现的接口的所有public方法
	CGlib:代理类继承自原始对象，如果代理类和原始不在一个包里面同时原始对象的方法时非public，那么这些非public方法就是不可见的。对于代理，所以事务失效
**解决：改为public**

## 循环引用

A依赖B，B依赖A

- spring的解决方法（非构造器注入）（三级缓存）

一级缓存：存放完成初始化的Bean
二级存放：已经实例化 的Bean,但是实行，依赖还是null
三级缓存：存放Bean的工厂（只要再实例化之后，就会产生一个工厂，将其放在里面）

A先实例化（此时已经将工厂放在了三级缓存）然后注入B,发现B没有就会去创建B,---->B实例化之后，注入A,先一级-->二级-->三级，最后再三级缓存，找到了A的工厂，通过A的工厂，生成一个半成品A,放入二级缓存，然后将这个半成品注入B,后面B就完成了初始化，放入一级缓存，然后，A此时就可以注入B了

## mybatis的延迟加载及原理（默认关闭）

如果一个user类，他由关联属性List<order>,此时如果只是去查询user的其他信息，实际也会去查询订单信息，再将其封装为完整的user,但有些时候我之只想知道user的其他信息，额外的关联sql就会造成性能浪费，延迟加载：当你需要这个List<order>的时候才会去查询

原理：当你查询user的其他信息时，这个通过JDK/CGLIB动态代理生成一个代理对象（什么也没有,有sql），当你调用getOrderList()的时候，mybatis就会通过sql将查询的内容封装成List<order>替换掉代理对象，此时user里面就会是真实的order了

## mybatis的一级，二级缓存

- 一级缓存(session)

**理解：基于sqlsession的，不同的sqlsession对应自己的缓存
比如mapper1是用sqlsession1创建的代理对象，当mapper1第一次查询getById(6),会查询数据库，并将其放入一级缓存（Hash）,后面如果还是这个session创建的代理，就会先看缓存是否存在，存在就不会插数据库，如果这两个代理对象不是同一个sqlssion创建的，那么就会查询

- 二级缓存(namespace)--->默认关闭

相当于是基于mapper,如果mapper1和mapper2都是一个mapper的代理对象，即使是不同的session,也具有同一个二级缓存

==注意：1.当session关闭后，一级缓存的东西会到二级缓存（如果没有开启二级缓存，就会被清空）（没有实现serilizable的不会转移）
		2.二级缓存缓存的数据须实现serilizable接口
        3.当你对数据库进行增删改的时候，二级缓存被清空==

## 同一条消息重复消费怎么解决（幂等性问题）？

当消费者正常处理完了消息，但是再发送确认的时候，网络波动较大，MQ由于没有收到确认，就会一直重复送这个消息，导致重复消费

**解决：**
1.给每个消息添加一个唯一标识，当消息到达消费者的时候，先判断这个id是否再存在与数据库（是否处理过），如果处理过就不处理了，反之则处理
2.锁机制，再处理之前，先判断要处理的数据是否已经被处理过了

- 无参构造
   elementdata指向一个空数组{}，==然后当第一次添加元素时，先扩容到10==，然后，后面当添加第11个元素的时候，就扩容到1.5倍,如果扩容到1.5倍仍然不够，则就会直接扩容到需要的容量（一般是添加collection集合的时候）

- 有参构造
当参数为0则让elementdata指向空数组
当参数小于0就抛出异常
当参数大于0，则就直接创建参数大小的数组

- 参数为collection集合
先创建一个Object[] a等于这个集合的toArray()

如果a的长度为0，就让elementdata指向空数组
如果a的长度大于0，先判断是否为ArrayList（因为ArrayList的toArray()返回的就是Object[],可以直接让elementdata指向），如果不是ArrayList,就用数组拷贝（创建一个新的Object[],将a的元素拷贝到这个数组上，再让elementdata指向这个数组）

## HashMap底层实现原理？

当添加元素的时候，通过key计算出一个hash值,这个hash值类似于数组下标，如果计算出来的这个hashcode，还没有元素加入，就直接将这个元素，加入其中，如果已经存在元素，判断key是否相同，如果key相同，就覆盖，反之就以链表的形式挂再下面（**不同的key也可能计算出相同的hashcode,不同的hashcode取模后，也有可能映射到相同的索引**），JDK1.7及以前，只有链表加红黑树，JDK1.7以后，当链表长度大于等与8，且数组长度大于等于64，就会转化为红红黑树（可以提高查询效率和插入的效率）

## HashMap的put流程

![](/images/notes/Screenshot 2025-12-09 161056.png)

先判断数组是否初始化,如果没有初始化，则初始化容量为16，然后通过hash函数，计算一个hashcode，根据hashcode mod 数组长度，找到这个key再数组中的存放位置，如果这个位置为null,说明还没有元素储存过，就直接将键值对储存再这里，如果非null,则先判断是红红黑树还是链表，如果是链表，则遍历链表，如果链表里面已经存在这个key,就覆盖，不存在的话就将这个key-value添加再链表尾部，如果是红红黑树，也是如此，添加完成之后，就判断（size+1是否大于阈值-->数组长度乘以加载因子），大于则扩容

##  HashMap扩容机制？

![](/images/notes/Screenshot 2025-12-09 164513.png)

先创建一个新的数组，大小为原来的两倍，遍历原数组的元素，通过如果是单个节点就直接根据他的hashcode&（新数组长度-1），计算出索引（因为新数组长度大小为2的幂，例如10000（16），减一后变成了01111，做与运算，高位全是0，相当于hashcode二进制的后四位做与运算,结果为0-15，所以这里的&和%效果相同，因为位运算更快，所以采用&），链表的话，就直接遍历链表，对链表的每一个节点，做上述运算，计算出新的位置(尾插)，会分裂成两个链表，红红黑树的话，计算出来再新数组的索引后，就按照红黑树添加规则添加元素

## 创建线程的方法？
**总结：**
1.要获取线程执行结果优先选择callable和线程池
2.线程池里面，既可以执行Runnable,也可以执行实现callable(返回值为futureTask)
- 继承Thread
```java
import java.util.concurrent.*;  
  
class myThread extends Thread {  
    @Override  
    public void run() {  
        System.out.println("Hello World");  
    }  
}  
  
public class Main {  
    public static void main(String[] args) throws ExecutionException, InterruptedException {  
        myThread t = new myThread();  
        t.start();  
    }  
}
```
- 实现Runnable接口
```java
import java.util.concurrent.*;  
  
class myThread implements Runnable {  
    @Override  
    public void run() {  
        System.out.println("Hello World");  
    }  
}  
  
public class Main {  
    public static void main(String[] args) throws ExecutionException, InterruptedException {  
        myThread t = new myThread();  
        Thread t1 = new Thread(t);  
        t1.start();  
    }  
}
```
- 实现callable接口
```java
import java.util.concurrent.*;  
  
class myThread implements Callable<Integer> {  
    @Override  
    public Integer call() {  
        return 1 + 1;  
    }  
}  
  
public class Main {  
    public static void main(String[] args) throws ExecutionException, InterruptedException {  
        myThread t = new myThread();  
        FutureTask<Integer> f = new FutureTask<>(t);  
        Thread t1 = new Thread(f);  
        t1.start();  
        System.out.println(f.get());  
    }  
}
```
- 线程池
```java
import java.util.concurrent.*;  
  
class myThread implements Callable<Integer> {  
    @Override  
    public Integer call() {  
        return 1 + 1;  
    }  
}  
  
public class Main {  
    public static void main(String[] args) throws ExecutionException, InterruptedException {  
        ExecutorService executorService = Executors.newFixedThreadPool(10);  
        Future<Integer> submit = executorService.submit(new myThread());  
        System.out.println(submit.get());  
        executorService.shutdown();  
    }  
}
```
